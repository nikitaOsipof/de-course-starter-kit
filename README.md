## 🚀 О проекте
Данный репозиторий является стартовым шаблоном для выполнения лабораторных работ. В течение семестра вы разработаете отказоустойчивый производственный конвейер (Data Pipeline) полного цикла. 

Репозиторий настроен как полноценный **`uv` проект**. Управление зависимостями, точные версии пакетов и настройки инструментов качества кода зафиксированы в связке файлов `pyproject.toml` и `uv.lock`. Это гарантирует 100% идентичность рабочего окружения у всех студентов и на сервере проверки.

## 🛠️ Инфраструктура и стек технологий
* **Среда и менеджер пакетов:** `uv` (Astral) — сверхбыстрый инструмент автоматизации на Rust.

* **Оркестрация и DevOps:** Docker, Docker Compose, Apache Airflow.
* **Качество кода (CI/CD):** Pytest, Unittest.mock, Ruff, Mypy.

## 📦 Инструкция по настройке окружения (`uv sync`)

Благодаря использованию `uv` в режиме проекта, вам не нужно вручную создавать и активировать окружение для установки библиотек.

1. **Установка `uv` на локальный компьютер:**
   * **Windows (PowerShell):** `irm https://astral.sh | iex`
   * **macOS / Linux:** `curl -LsSf https://astral.sh | sh`

2. **Автоматическая сборка проекта:**
   Откройте терминал в папке проекта и выполните всего одну команду:
   ```bash
   uv sync --all-groups
   ```
   *Что произойдет:* `uv` сам создаст скрытую папку `.venv`, скачает изолированный Python 3.11 и установит все базовые и dev-библиотеки курса строго в соответствии с файлом-замком `uv.lock`.

3. **Запуск скриптов:**
   Для запуска скриптов или тестов используйте команду `uv run`. Она автоматически использует созданное окружение без необходимости его ручной активации:
   ```bash
   uv run generate_fixtures.py  # Запуск генератора данных
   uv run pytest tests/         # Запуск тестов
   ```

4. **Добавление новых библиотек:**
   Если вам потребовался новый пакет, выполните команду:
   ```bash
   uv add <название_библиотеки>
   ```
   `uv` автоматически установит пакет и внесет изменения в `pyproject.toml` и `uv.lock`.

## 📋 Технические задания к лабораторным работам

### Раздел 1. Продвинутый Python и инженерная надежность

#### ЛР №1: Продвинутое ООП и контекстные менеджеры
* **Файл:** `src/connector.py`
* **Задача:** Реализовать класс `SQLiteDataConnector` для инкапсуляции логики работы с базой данных SQLite.
* **Требования:** Реализовать дандер-методы `__enter__` и `__exit__` для поддержки протокола `with`. Метод `__exit__` должен автоматически делать `commit` при успехе и `rollback` при любом исключении. Все SQL-запросы должны быть параметризованными (защита от SQLi).

#### ЛР №2: Потоковая обработка данных и Генераторы
* **Файл:** `src/stream_reader.py`
* **Задача:** Написать функцию-генератор `log_stream_reader(file_path)` для ленивого чтения лог-файла `data_samples/app_logs.txt`.
* **Требования:** Использовать ключевое слово `yield`. Скрипт должен построчно читать файл, преобразовывать строки в словари и отдавать их наружу только при соответствии уровню `ERROR`. Память процесса (RAM) не должна расти при обработке файла любого объема.

#### ЛР №3: Декораторы и отказоустойчивость пайплайнов
* **Файл:** `src/decorators.py`
* **Задача:** Написать параметризованный декоратор `@retry(max_tries=3, delay=2)` для обработки нестабильных сетевых соединений.
* **Требования:** Декоратор должен перехватывать ошибки `sqlite3.OperationalError` и `psycopg2.OperationalError`. При сбое скрипт должен делать паузу `time.sleep` и повторять вызов. После `max_tries` попыток — выбрасывать ошибку выше.

### Раздел 2. Обработка, валидация и хранение данных

#### ЛР №4: Валидация данных с Pydantic и ORM SQLAlchemy
* **Файл:** `src/models.py`
* **Задача:** Создать шлюз валидации "грязных" данных из `data_samples/raw_orders.json` перед их сохранением в PostgreSQL.
* **Требования:** Описать схему данных через `pydantic.BaseModel` (поля `id`, `timestamp`, `status`, `amount` > 0, статус строго из списка `['new', 'processing', 'done']`). Описать структуру таблицы через `SQLAlchemy`. Валидные строки пишутся в БД, невалидные — сохраняются в файл `rejected_rows.json`.

#### ЛР №5: Аналитическая обработка данных с Polars
* **Файл:** `src/analytics.py`
* **Задача:** Реализовать аналитический скрипт расчета витрины данных на базе библиотеки Polars.
* **Требования:** Использовать ленивый режим вычислений (`pl.scan_csv`). Написать цепочку выражений: фильтрация аномалий, группировка, расчет агрегатов (среднее, медиана) и расчет доли строки внутри группы через оконную функцию `.over()`. Результат сохранить в бинарный колоночный формат `.parquet`.

### Раздел 3. Промышленная инфраструктура и оркестрация (Production Ready)

#### ЛР №6: Контейнеризация ETL-скрипта в Docker
* **Файл:** `Dockerfile`
* **Задача:** Упаковать разработанный Python-скрипт в Docker-контейнер.
* **Требования:** Написать `Dockerfile` по принципу **Multi-stage build** на базе образа `python:3.11-slim`, чтобы отсечь dev-зависимости (`pytest`, `ruff`, `mypy`) и оставить образ легковесным. Все реквизиты доступа к БД должны считываться строго из переменных окружения (`os.getenv`), передаваемых через локальный файл `.env`.

#### ЛР №7: Оркестрация пайплайнов в Apache Airflow
* **Файл:** `airflow/dags/orders_etl_dag.py`
* **Задача:** Написать сценарий автоматизации и запуска конвейера по расписанию (DAG).
* **Требования:** Развернуть стек Airflow + PostgreSQL через `docker compose up -d`. Описать граф задач: создание таблицы (`PostgresOperator`) -> выгрузка файла (`PythonOperator`) -> валидация и импорт (`PythonOperator`). Задать расписание на ежедневный запуск (`schedule_interval='@daily'`).

#### ЛР №8: Тестирование пайплайнов (Pytest и Mock)
* **Файл:** `tests/test_etl_logic.py`
* **Задача:** Покрыть критические узлы трансформации данных модульными тестами.
* **Требования:** Выделить функции очистки данных в чистые функции и протестировать их граничные условия (пустые строки, `None`) через `pytest`. Используя `unittest.mock.MagicMock`, полностью изолировать слой записи в базу данных, проверяя факт вызова методов без реальных сетевых запросов.

## 🚦 Правила сдачи работ и автоматическая проверка (CI/CD)

1. Каждая лабораторная работа выполняется в **отдельной Git-ветке** (`feature/lr1`, `feature/lr2` и т.д.).
2. Сдача работы происходит путем создания **Pull Request** из вашей ветки в ветку `main`.
3. При создании Pull Request на GitHub автоматически запускается настроенный робот (GitHub Actions):
   * Он проверяет ваш код линтером `ruff` на соответствие стандартам стиля PEP 8.
   * Он проверяет типы данных через статический анализатор `mypy`.
4. **Если код грязный или содержит ошибки типов** — пайплайн загорится красным крестиком. Такие работы к ручной проверке преподавателем **не допускаются**. Исправляйте замечания линтера локально и отправляйте новые коммиты в вашу ветку.
